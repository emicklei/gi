# AI Agent Guidelines for `gi` (Go Interpreter)

Welcome! This document provides guidelines and details about the architecture, coding style, and testing practices for the `gi` project. `gi` is a Go interpreter written in Go, which parses Go source code, converts it to a Control Flow Graph (CFG), and executes it via an internal Virtual Machine (VM) using the Go reflection API.

---

## 1. Project Architecture

The interpreter works in distinct phases, largely contained within the `pkg/` directory:

1. **AST Parsing & Building**:
   - The project builds an intermediate representation of Go source code.
   - `ast_builder.go` visits Go standard AST nodes and converts them into an internal tree representation (expressions, statements, declarations).

2. **Control Flow Graph (CFG)**:
   - Statements and expressions are mapped to executable `Step` structures (found in `step.go` and `graph_builder.go`).
   - Steps form a directed graph where each `Step` points to a `Next` step, or multiple steps (e.g., `conditionalStep` for if/else flows).

3. **Virtual Machine (`vm.go`)**:
   - Operates by walking through the linked `Step` nodes.
   - Uses a Call Stack (`stack.go`, `stackframe.go`).
   - Operands and evaluation results are pushed to and popped from a frame's operand stack as `reflect.Value`.

4. **Environment & Memory**:
   - **Environment (`env.go`)**: Manages block-scoped variable declarations, storing variables as `reflect.Value`.
   - **Heap (`heap.go`)**: Used to represent dynamically allocated values and pointers.

5. **Debugging & DAP (`pkg/dap/`, `dap_access.go`)**:
   - `gi` includes a Debug Adapter Protocol (DAP) server capable of stopping at breakpoints, stepping, and inspecting environments.

---

## 2. Directory Structure

- **`pkg/`**: The core logic of the interpreter. Contains AST representations (e.g., `if.go`, `binary.go`), graph building, VM logic, stdlib bindings, and evaluation execution files.
- **`cmd/`**: Contains binaries related to the project:
  - `gi/`: The command-line interface for running and debugging scripts.
  - `genstdlib/`: A tool to generate standard library wrapper logic (outputting to `pkg/stdlib_*.go`).
  - `treerunner/`: Used to run Go programs sequentially, typically checking `gobyexample` code.
- **`examples/`**: Standalone Go scripts meant to demonstrate specific features of the interpreter.
- **`gobyexample/`**: Submodule or working directory (pulled during testing) used to test `gi` against a large corpus of standard Go snippets.

---

## 3. Code Style and Conventions

- **Go Idioms**: Standard Go formatting (`gofmt`), minimal stuttering, and straightforward interfaces are expected.
- **`reflect.Value` Heaviness**: Because the VM interprets a statically typed language dynamically, `reflect.Value` is pervasive. Handle invalid or untyped reflection values with care (e.g., `reflectUndeclared` or `reflectNil`).
- **Encapsulation**: State structures like `stackFrame`, `vm`, `env`, and `step` mostly keep fields private (`lowercase`) to tightly control state mutation during execution.
- **Diagnostics vs Completeness**: When resolving bugs or handling diagnostics, provide complete and semantically correct code rather than simplifying implementations just to make type checkers pass.

## 4. Testing Guidelines

Tests live alongside the code in the `pkg/` folder (e.g., `programs_test.go`, `assign_test.go`). 

- **Execution Testing**:
  Most tests are integration-style checks that parse small Go programs, feed them to the VM, and capture standard output to verify correctness.
  - Use `testMain(t, "source code", "expected output")` available in `programs_helper_test.go`.
- **Debugging Test Failures**:
  - Environment variables `GI_TRACE=1` and `GI_CALL=out.dot` are incredibly useful. 
  - `GI_TRACE=1` traces frame stack allocations, pushed/popped operands, and step execution.
  - `GI_CALL` visually maps the built AST and Control Flow Graph using Graphviz. 
- **Parallelism**: 
  Tests in `pkg` are highly parallelized (configured in `Makefile` with `go test -p 16`). Make sure test helpers that rely on global mutability (like overriding `print` functions) handle thread safety or use the internal `collectPrintOutput` safely.
- **Running Tests**:
  - `make test`: Runs all `pkg` tests in parallel.
  - `make serial`: Runs tests sequentially with tracing and call graphs generated.
  - `make gobyexample`: Runs a massive suite of external tests from the Go By Example repository.

---

## 5. Maintenance and Tooling

- **Linters**: The project enforces strict linting using `golangci-lint` (`make lint`) and checks for dead code (`make dead`). Ensure any newly added code is utilized or correctly exported.
- **Code Generation**: Before modifying `stdlib` behaviors, check if they are generated by `cmd/genstdlib`. You may need to modify the generator rather than the output files in `pkg/`.
- **Dependencies**: Use minimal external dependencies. Most logic should rely heavily on the Go standard library, specifically `go/ast`, `go/parser`, `go/token`, and `reflect`.

When assigned a task, favor creating focused tests in existing test files to verify your changes, utilize the VM trace tools for debugging internal operand stack issues, and keep the step execution lightweight.
